# Frequently asked questions

The fast answers. For the long-form architecture and rules, [`CLAUDE.md`](https://github.com/nyuchitech/design-portal/blob/main/CLAUDE.md) is the single source of truth.

## Getting started

### How do I install a single component?

```bash
npx shadcn@latest add https://design.nyuchi.com/api/v1/ui/button
```

The shadcn CLI copies the source into your repo. You own and edit it freely. To install several at once, list multiple URLs on the same `add` command. Browse the live registry at [/components](/components).

### How do I bootstrap a brand-new app on the design system?

If you're using Claude Code, type `/ecosystem-app-setup` — the skill walks the model through the eight-step setup (Next.js + Tailwind 4 + shadcn init + theme tokens + typography + `next-themes` + first components + theme provider). Without Claude Code, follow [`docs/installation`](/docs/installation) end to end.

### Do I need a Supabase account to use the portal?

No. The shadcn install path is fully public. You only need Supabase credentials if you want to run the portal **itself** locally — the `/api/v1/*` routes and the MCP server read from Supabase, and the dev server returns a clean 503 if the env vars aren't set. Set `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local` if you want everything live.

### Why don't I see a "Tailwind config" file?

The system uses Tailwind CSS 4's `@theme inline` block inside `app/globals.css` — there's no `tailwind.config.js`. All design tokens (Five African Minerals, semantic colors, radii, fonts) live as CSS custom properties in `globals.css` and are registered for utility-class generation in the `@theme` block. Copy the `:root`, `.dark`, and `@theme inline` blocks verbatim into a new app.

## The 3D frontend architecture

### What does "ten layers across five axes" actually mean?

The frontend is modelled in three primary dimensions plus two outliers:

- **X-axis — horizontal composition flow** (what the user sees): L2 primitives → L3 brand → L6 pages → L7 shell.
- **Y-axis — vertical infrastructure** (threading through every X-layer): L1 tokens, L4 safety, L5 resilience.
- **Z-axis — depth observation** (watching X and Y from above without being inside anything): L8 assurance.
- **Outside — actors beyond the build**: L9 fundi (self-healing).
- **Documentation — the system describing itself**: L10.

Components import from the layer below on the same axis — never sideways or upward. Full table with covenants in [`CLAUDE.md` §6.2](https://github.com/nyuchitech/design-portal/blob/main/CLAUDE.md#62-layered-component-architecture) and on the [/architecture](/architecture) page.

### Why are fundi (L9) and documentation (L10) separate axes?

Because they don't compose into the product — they keep the product from going stale. Both run continuously, outside any release cadence. See the "Resilient by design" section on the [landing page](/) and the [Fundi guide](/architecture/fundi).

### Is this the same as the 7-layer data architecture I've seen?

No, and never conflate them. The 3D model above is the **frontend** architecture (10 layers × 5 axes). The data architecture under [/architecture](/architecture) (Pod → Relational → Document → Orchestration → Edge → Device → Open Data) is a separate 7-layer model describing how data moves through the ecosystem. A component can sit at frontend L3 and read from data L2 without contradiction.

### Why are there sometimes more components in the registry than what I see on disk?

Only the small set of primitives the portal **itself** imports gets committed under `components/ui/`. The other registry items live exclusively in the Supabase `components` table and are served on demand via `/api/v1/ui/{name}`. `registry.json` is a CI-verified snapshot, not a source of truth — `pnpm registry:sync` regenerates it from Supabase.

## The MCP server

### What does the MCP server do that the API doesn't?

The MCP server exposes the same data as `/api/v1/*` but as **callable tools and resources** for AI assistants (Claude Code, Cursor, Copilot, etc.). 18 tools and 5 resources let your AI install components, scaffold new ones, look up tokens, read live docs, and inspect the architecture without you having to copy-paste URLs. Connect at `https://design.nyuchi.com/mcp` (Streamable HTTP).

### How do I add the MCP server to Claude Code?

Add to your `.claude/settings.json`:

```json
{
  "mcpServers": {
    "nyuchi-design-portal": {
      "type": "url",
      "url": "https://design.nyuchi.com/mcp"
    }
  }
}
```

Restart Claude Code. The 18 tools and 5 resources show up in the next session.

### Where do the answers come from when I ask the AI a question?

Live Supabase. The MCP server's system prompt is loaded from the `ai_instructions` table at startup with a 60-second TTL cache, so updates to doctrine roll out with zero deploy. Tools that surface counts, versions, or registry contents query the DB on every call — no hardcoded numbers anywhere.

## Fundi self-healing

### Who can report issues to fundi?

The L8 assurance layer in any consuming app. Fundi reporting is **on by default** for every Nyuchi-owned (`*.nyuchi.com`, `nyuchi.africa`) and Mukoko-owned (`*.mukoko.com`) domain, plus `dev` environments and allow-listed staging. External customers consuming the registry are strictly **opt-in** via `NEXT_PUBLIC_FUNDI_ENABLED=1`. Full deployment matrix in `supabase/functions/README.md`.

### What happens to a fundi report after it's submitted?

`POST /functions/v1/fundi` validates and writes to `fundi_issues` (deduped by FNV-1a fingerprint). Every 15 minutes the cron-driven `POST /fundi/heal` pass — auth-gated by Supabase service-role key or `FUNDI_HEAL_TOKEN` — picks up the worst severity-priority open issues, opens corresponding GitHub issues on `nyuchi/design-portal`, and writes to the append-only `fundi_healing_log`.

### Can attackers spam the fundi ingest?

Schema validation is strict (`scope` matches `/^[a-z0-9._-]{1,40}$/i`, `symptom` ≤ 200 chars, `severity` enum-validated), every report is deduped by fingerprint, and the heal endpoint is auth-gated. The worst an unauthenticated caller can do via ingest alone is fill the `fundi_issues` table with rows that never get promoted.

## How does the system stay current?

### How do you stop the registry, the docs, and the AI tools from drifting apart?

Single source of truth in Supabase, no hardcoded counts anywhere, CI verifies `registry.json` matches the DB on every PR (`pnpm registry:verify`), and the no-deferral security rule (CLAUDE.md §15 rule 22) means findings get fixed in the PR that surfaces them — never punted to a follow-up issue.

### How do I run the same checks CI runs before I push?

```bash
pnpm check
```

That single command chains every CI gate (`format:check && lint && lint:md && lint:json && typecheck && test && audit:check && registry:verify && build`) and exits non-zero on the first failure. See [`CONTRIBUTING.md`](https://github.com/nyuchitech/design-portal/blob/main/CONTRIBUTING.md) for the one-time tooling setup.

### What's the policy when a security audit turns up a vulnerability?

Fixed in the **same PR** that surfaced it, regardless of original scope. From CLAUDE.md §15 rule 22:

> Any vulnerability surfaced during a `/security-review`, manual audit, CodeQL alert, Dependabot advisory, or `pnpm audit` must be fixed in the current PR — even if the original PR scope is "docs only" or "feature X". Do not file a follow-up issue and merge the unfixed code.

The only exception is when the fix concretely depends on infrastructure that isn't on the branch (e.g. a Supabase migration); even then every code-level mitigation that doesn't require the missing infrastructure ships in the PR. Tracked in `SECURITY.md`.

## Brand & doctrine

### Why are all the brand wordmarks lowercase?

Doctrinal: `mukoko`, `nyuchi`, `shamwari`, `bundu`, `nhimbe`. Lowercase is the canonical typeset across the ecosystem. Uppercasing them is a brand violation flagged on PR review.

### Why are buttons always pill-shaped?

Executive brand identity decision. All buttons, tabs, and pill-shaped controls use `rounded-full` (9999px). Not a radius-scale value to override per app — it's an ecosystem invariant.

### Why 56px touch targets?

The portal serves the African mobile market: shared devices, outdoor use, varied screen sizes, accessibility. 56px is the default; 48px is the minimum (small variants only). Non-negotiable per CLAUDE.md §8.2.

### What's APCA and why don't you use WCAG 2 ratios?

APCA (Advanced Perceptual Contrast Algorithm) is the WCAG 3 contrast standard. It correctly weighs perceived contrast based on the human visual system, where WCAG 2's pure luminance ratio over- or under-rates many real-world combinations. Body text targets Lc 90+, large text Lc 75+, UI labels Lc 60+. Pre-computed mineral Lc values are in the `nyuchi-design-system` skill.

### Where do "Five African Minerals" come from?

Five mineral colors that are constant across light and dark mode: Cobalt, Tanzanite, Malachite, Gold, Terracotta. Each maps to a CSS custom property (`--color-cobalt`, etc.) and a Tailwind utility (`bg-mineral-cobalt`). Full palette and category-to-mineral mapping under [/brand](/brand).

## Versioning, releases, and deploys

### How do I release a new version?

Bump `package.json`, `lib/mcp-server.ts`, and `components/landing/footer.tsx` to the new number, insert a row into the Supabase `changelog` table, then commit and tag. The release workflow auto-validates and creates the GitHub release. Full process in CLAUDE.md §14.

### Where can I see what changed recently?

`GET /api/v1/changelog` is the live machine-readable feed. The `/docs/changelog` page renders the same data. The repo's `CHANGELOG.md` carries the unreleased section and historical detail.

### Can I deploy the portal myself?

Yes — Vercel is the canonical host but the app is a stock Next.js 16 build. You'll need the Supabase env vars set in the host. The MCP server is a Next.js API route (`app/mcp/route.ts`), so it ships in the same deploy.
