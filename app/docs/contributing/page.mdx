# Contributing

The Nyuchi Design System is open source and built by the community. Ubuntu — I am because we are.

## Repository

github.com/nyuchi/design-portal

## Source code lives in the database

Production source for every stable component is stored in `components.source_code` in Supabase (project `grjsboqkaywpwatvrzmy`). The frontend repo pulls from the database at build time — it does not maintain a separate source-of-truth in the file system. To add or modify a component, write the source directly to `components.source_code` via SQL.

## Adding a New Component

1. Decide which layer it belongs to (see Layer Decision Guide)
2. Create the component following the layer enterprise criteria
3. Insert into the components table with status=alpha (registered, no source yet)
4. Write the production TypeScript into components.source_code
5. Flip status to stable when source is production-ready
6. Add component_docs entry with use_cases, variants, features, a11y notes
7. Insert a component_demos manifest row (has_demo=true, demo_type=interactive or code-example)
8. Ensure data-slot and data-portal attributes are present in the source
9. Log the version with log_version(name, type, comment, author)
10. Submit a PR

## Enterprise Criteria Checklist

### Layer 2 (Primitives)
- [ ] No useNyuchiHarness import
- [ ] data-slot attribute on root element
- [ ] data-portal attribute pointing to design.nyuchi.com/components/\{name\}
- [ ] cn() for className composition
- [ ] Zero raw Tailwind colors
- [ ] Import icons from @/lib/icons
- [ ] Touch targets follow brand_touch_targets (48px minimum)
- [ ] If pill-shaped category (button, input, avatar, badge, toggle), use borderRadius 9999

### Layer 3 (Brand)
- [ ] nyuchi- prefix on the name
- [ ] useNyuchiHarness with full destructure \{ log, motion, LiveRegion \}
- [ ] animStyle with motion.prefersReduced check
- [ ] ARIA role or aria-label
- [ ] data-slot and data-portal attributes
- [ ] focus-visible ring on interactive elements
- [ ] min-h-[48px] touch targets on buttons (brand_touch_targets.comfortable)
- [ ] I18N (Intl formatters for dates, numbers, currency)
- [ ] Semantic tokens for status-bearing colors

### Layer 6 (Pages)
- [ ] Pure composition — no inline buttons, cards, or SVGs
- [ ] Accept children/slots for content
- [ ] Semantic CSS vars only (bg-card, text-foreground, bg-primary)
- [ ] Loading state prop
- [ ] role="main" and aria-label
- [ ] data-slot and data-portal

### Layer 8 (Assurance)
- [ ] TypeScript module exporting typed functions and React hooks
- [ ] Banner comment identifying layer and purpose
- [ ] Configurable rules array
- [ ] Optional onViolation/onComplete callbacks
- [ ] React hook variant for continuous monitoring

## Accessibility Validation

Before shipping a new component, validate text-on-background color combinations:

```sql
SELECT calculate_contrast_ratio('#FFFFFF', '#0047AB');
-- Returns the WCAG 2.1 contrast ratio. AA needs 4.5, AAA needs 7.0.
```

For systematic pair tracking, look at `brand_accessibility_checks`. Every critical pair already has a row with computed ratio and color-blindness flags.

## RTL Support

Use logical CSS properties (margin-inline-start, padding-inline-end, inset-inline-start) not physical (margin-left, padding-right, left). Use text-align: start/end not left/right. Mark directional icons with data-rtl-mirror="true". The rtl-conformity-check L8 component validates these rules at runtime.

## Versioning

Use the log_version function:

```sql
SELECT log_version('button', 'fix', 'Fixed focus ring on disabled state', 'your-github-handle');
```

Change types: created, fix, enhancement, tokens_wired, layer_moved, breaking, deprecated, docs.

## Fundi Integration

When L8 detects an issue with your component, Fundi creates a GitHub issue with structured labels. You can find issues related to your component at:

github.com/nyuchi/design-portal/issues?q=label:fundi:layer/\{your-layer\}
