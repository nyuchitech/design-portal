# API Reference

The Nyuchi Design Portal exposes a REST API and MCP server for accessing the component registry.

## Base URL

```
https://design.nyuchi.com/api
```

## Endpoints

### Components

| Method | Endpoint | Description |
|---|---|---|
| GET | /api/components | List all components |
| GET | /api/components/\{name\} | Get component by name |
| GET | /api/components/\{name\}/source | Get source code (lives in components.source_code in DB) |
| GET | /api/components/\{name\}/docs | Get documentation |
| GET | /api/health/\{name\} | Health check |
| GET | /api/changelog/\{name\} | Version history |

### Search

| Method | Endpoint | Description |
|---|---|---|
| GET | /api/search?q=\{query\} | Search components |
| GET | /api/search?category=\{cat\} | Filter by category |
| GET | /api/search?layer=\{n\} | Filter by layer |
| GET | /api/search?status=\{status\} | Filter by status (stable, alpha, deprecated) |

### Documentation

| Method | Endpoint | Description |
|---|---|---|
| GET | /api/docs | List documentation pages |
| GET | /api/docs/\{slug\} | Get page by slug |

### Fundi

| Method | Endpoint | Description |
|---|---|---|
| POST | /api/chaos/\{name\} | Inject chaos (L8) |
| POST | /api/fundi/\{name\} | Trigger healing (L9) |
| GET | /api/fundi/issues | List open healing issues |

## MCP Server

```
https://design.nyuchi.com/mcp
```

Tools: list_components, search_components, get_component, get_component_docs, get_install_command, get_design_tokens, get_brand_info, get_architecture_info

## SQL Functions (Canonical Counts)

The database exposes count functions that should be used instead of hardcoded numbers anywhere they appear:

| Function | Returns |
|---|---|
| `get_layer_counts()` | per-layer component counts with axis and label |
| `get_system_counts()` | system-wide totals in one row |
| `get_token_counts()` | row counts for every brand_* table |
| `get_layer_categories(N)` | category breakdown within layer N |
| `get_latest_release()` | most recent changelog entry |
| `get_accessibility_summary()` | WCAG compliance and color-blindness validation summary |

## SQL Functions (Accessibility Validation)

Three functions support contrast-ratio validation directly in the database:

| Function | Returns |
|---|---|
| `hex_to_rgb(hex)` | (r, g, b) integer tuple |
| `relative_luminance(hex)` | WCAG relative luminance numeric value |
| `calculate_contrast_ratio(fg_hex, bg_hex)` | WCAG 2.1 contrast ratio (1.00 to 21.00) |

## SQL Functions (Component Lifecycle)

| Function | Purpose |
|---|---|
| `log_version(name, type, comment, author)` | Track component changes |
| `log_release(version, title, desc, layers, axes, added, modified, deprecated, author)` | Log a release to the changelog |
| `get_component_links(name)` | Get all canonical URLs for a component |
| `get_component_history(name, limit)` | Component version history |
| `create_fundi_issue(component, severity, error_type, source, payload)` | Create healing issue |
| `is_domain_allowed(domain, mode)` | Check observability allowlist |

## Direct Database Tables

When you need raw token values, query the `brand_*` tables directly. The full list is available via `get_token_counts()` and includes:

Visual tokens: brand_minerals, brand_semantic_colors, brand_spacing, brand_typography, brand_radius, brand_shadow, brand_motion, brand_breakpoints, brand_z_index.

Sizing & accessibility: brand_icon_sizes, brand_touch_targets, brand_component_specs.

Density: brand_density, brand_density_breakpoint_map.

Validation: brand_accessibility_checks (per-pair WCAG and color-blindness flags).

Reference: brand_categories, brand_ecosystem, brand_meta.
