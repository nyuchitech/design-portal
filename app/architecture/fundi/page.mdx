# Fundi — Self-Healing Intelligence

Fundi (Swahili: fixer, mechanic, technician) is the outside actor that heals the system. It operates through GitHub Issues, keeping humans in the loop.

## Architecture Flow

```
L8 detects failure
    ↓
nyuchi-fundi-reporter creates GitHub issue
    → Labels: fundi:severity/high, fundi:layer/5, fundi:type/render
    → Body: Component portal link, blast radius, diagnostic
    ↓
GitHub webhook → fundi-webhook edge function
    ↓
Fundi diagnoses → comments on issue
    → "Isolated render error. Cache-clear (auto-fixable). Escalate to owner (needs human)."
    ↓
Human reviews on GitHub
    ↓
nyuchi-fundi-learning records outcome
```

## GitHub Labels

| Label | Purpose |
|---|---|
| fundi:severity/\{low,medium,high,critical\} | How urgent |
| fundi:layer/\{1-8\} | Which architecture layer |
| fundi:type/\{render,network,data,auth,chain,crypto,timeout,a11y,perf,conformity,slo\} | Error classification |
| fundi:source/\{chaos,synthetic-probe,error-tracker,...\} | Which L8 component detected it |
| fundi:auto-fixable | Fundi can fix without human |
| fundi:awaiting-review | Human review needed |

## Decision Rules

- L1 (Tokens) or L4 (Safety) failure → always escalate, never auto-fix
- Auth errors → always escalate
- Network/timeout → auto-fixable (retry + cache-clear)
- Conformity violation → auto-fixable (registry sync)
- Performance regression → human review needed
- Systemic blast radius → emergency degradation + escalate

## Database Tables

- fundi_issues: tracks every failure from detection through resolution
- fundi_healing_log: logs individual healing actions with success/failure
