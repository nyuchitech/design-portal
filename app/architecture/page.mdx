# The Nyuchi 3D Architecture

The Nyuchi Design System is not a flat stack. It is a three-dimensional model where components exist at intersections of axes. Understanding the geometry is the key to knowing where any given component belongs.

## Why three dimensions?

Most component libraries are flat: primitives at the bottom, composed components above, pages on top, with some infrastructure bolted on the side. This works for small systems but fails for super-apps, because super-apps need to express concepts that don't fit a linear stack. Where does "safety" belong? It's not above pages, and it's not below primitives — it runs *through* every layer. Where does "observability" belong? Same answer — it watches everything without being inside anything.

The 3D model solves this by giving each kind of concern its own dimension.

## X-axis — Horizontal composition (what users see)

The X-axis is the traditional component composition stack. Users interact with these layers directly. Each layer builds on the one below it:

- **L2 Primitives** — raw building blocks like button, card, input, dialog, segmented-control, toolbar, bento-grid, inline-edit. Unbranded, generic, could work in any product.
- **L3 Brand** — Nyuchi-specific compositions of primitives, carrying the mineral palette and harness integration. These are what make your app *look* like a Nyuchi app.
- **L6 Pages** — full-screen compositions that arrange L2 and L3 components into complete experiences.
- **L7 Shell** — the app container that holds pages: navigation, routing, lifecycle, theme provision, global overlays.

Think of building a house: bricks (L2) → painted bricks (L3) → rooms (L6) → doors and roof (L7).

## Y-axis — Vertical infrastructure (what flows through)

The Y-axis is the infrastructure that runs through every X-axis layer like plumbing and electrical running through every room of a house:

- **L1 Tokens** — the mathematical substrate (colours, spacing, radius, shadow, motion, breakpoints, z-index, icon sizes, touch targets, density, and the brand_accessibility_checks validation table). Every other layer reads from here.
- **L4 Safety** — conditional-rendering gates that decide who sees what (permission, geography, rate limit, wallet state, content moderation).
- **L5 Resilience** — the failure-case handlers (error boundaries, skeletons, offline banners, fallback chains).

None of these are things a user sees directly, but every component they see is built on top of them.

## Z-axis — Depth observation (what watches)

The Z-axis is the assurance layer — instrumentation that watches the entire 3D space without being part of it:

- **L8 Assurance** — chaos injection, synthetic probes, performance monitoring, error tracking, incident management, conformity checking. Includes accessibility audit (`nyuchi-a11y-audit`), RTL conformity check (`rtl-conformity-check`), API probes, alert engine, and SLO tracking. This is your Netflix Conformity Monkey layer plus observability plus incident response.

## Outside actors

Two layers are described as *outside actors* — they act on the system without being inside it:

- **L9 Fundi** — self-healing that responds to L8 detections by creating GitHub issues and, where appropriate, auto-fixing issues. Fundi is Swahili for "fixer/mechanic/technician," which is exactly what this layer does.
- **L10 Documentation** — this documentation layer. Guides, AI instruction sets, docs engine, changelog renderer.

## Source code lives in the database

Every stable component carries its production TypeScript in `components.source_code`. The frontend repo pulls from the database at build time rather than maintaining a separate source-of-truth in the file system. To add or modify a component, write the source directly to the database via SQL and flip its status from `alpha` to `stable`.

## Finding current component counts

This page deliberately does not state how many components live in each layer, because those counts drift as the system evolves. To see current counts, query the database:

```sql
SELECT * FROM get_layer_counts();
```

This returns one row per layer with component_count, stable_count, alpha_count, and deprecated_count. It also includes the axis and layer_label for context. For category breakdowns within a specific layer, use `get_layer_categories(N)` where N is the layer number. For token category counts across every brand_* table, use `get_token_counts()`. For accessibility compliance, use `get_accessibility_summary()`.

## The consistency guarantee

Each layer has exactly one sub-label value in the `components.layer` column. This means a query filtered by architecture_layer and a query filtered by the sub-label return identical results:

| Layer | Sub-label |
|---|---|
| L1 | tokens |
| L2 | primitive |
| L3 | brand |
| L4 | safety |
| L5 | resilience |
| L6 | pages |
| L7 | shell |
| L8 | assurance |
| L9 | fundi |
| L10 | documentation |

A developer filtering "show me Layer 5 resilience" versus "show me all resilience components" gets the same set, because the sub-labels agree.
